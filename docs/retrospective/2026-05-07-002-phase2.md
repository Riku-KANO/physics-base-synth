## 4. 躓きと教訓

### 4.1 C8 ピッチ精度の物理限界（最大の学び）

仕様書 D14 / F12 は「A1〜C8 ±0.5% 以内」を要求していたが、Step 7 でテストを書いて初めて以下が発覚:

- 整数 τ の autocorrelation は C8 (周期 11.5 サンプル) では 4-9% 誤差。`parabolic interpolation` で sub-sample 精度に持っていく必要がある（仕様書 06 章で先回りされていた）
- ところが parabolic 補正以前の問題で、C8 の信号自体に C8 fundamental の周期性が見えない。autocorr が monotonically decreasing
- 原因: Lagrange 3 次補間の周波数応答 `|H_lag(C8)| ≈ 0.998`（length_frac=0.466 の場合）。loop gain `damping × |H_lag|` が < 1 になり C8 の自己発振条件 (loop gain ≥ 1) を満たさない
- DC は `|H_lag(0)| = 1.0` で sustain するため、初期励振直後は C8 周波数の AC 成分があるが急速に減衰し、0.1 秒スキップ後の解析窓では DC ドリフトが支配的になる
- これは KS の架空的な構造 (FIR Lagrange + IIR brightness LPF + damping ループ) の合成的な振る舞いで、実機 KS 弦シンセに共通する高音域の限界

**得た教訓:**

- **仕様書がフォールバックを用意していた**: 06 章 R23 (5)「どうしても不安定なら C8 のみ許容誤差を緩和し test_pitch_c6 を主検証として残す」をそのまま採用、 `#[ignore]` 化と doctest 文での説明に着地
- **物理シミュレーションは仕様書策定段階では限界が見えにくい**: 5 ラウンドのレビューを経た仕様書でも、Lagrange の周波数応答が loop gain を下げる効果は事前計算されていなかった
- **テストを書くまで気付けない**: 仕様策定段階で C6 の周期 ~46 サンプルと C8 の周期 ~11.5 サンプルが質的に違うことが分かる手段はなく、実装してテストして初めて見える
- **Phase 3 で Thiran allpass 検討の余地**: 補間が IIR でも `|H| = 1` を保つので C8 fundamental が sustain する可能性。ただし IIR の極配置が新たな安定性議論を生む

### 4.2 mono mode が一発で書けなかった（D29）

- Step 13 の最初の note_off 実装は「new top があれば revive、空なら note_off」で済ませた → D が release されないバグ
- Step 14 のテスト追加で初めて顕在化
- 修正後、review で「MIDI 重複 noteOn で stale 履歴が残る」もう 1 件発覚 → `push_unique` 導入
- mono mode は「現在鳴っているボイスを止めるタイミング」「履歴に重複させない」「中間キー解放で再 trigger しない（クリック対策）」の 3 軸を独立に実装する必要があり、ワン・パスでは正しく書けなかった

**得た教訓:** mono synth の挙動は intuitive に書くと罠が多い。テスト先行（特に「重複 noteOn / 中間解放 / 17 鍵 overflow」のような edge case）が必要。

### 4.3 const generic と const ハードコード定数の不整合（D20）

Step 11 で `POLY_SCALE = 0.353_553_4` を const として書いた → レビューで「VoicePool<4> や VoicePool<16> で gain 誤りが silent に入る」と指摘 → `new()` で計算する `poly_scale: f32` field 化に修正。

**得た教訓:** const generic を持つ型のパラメータは、N に依存する定数も generic に解く必要がある。`f32::sqrt` が const fn ではないので、コンパイル時計算は不可能で `new()` 計算しかない。これは Rust の制約。

### 4.4 ホットパスの剰余演算

Simplify レビューで「buf_len は power-of-two ではないので LLVM は `%` をマスク化できない」と指摘され、Lagrange 4 read 位置の `%` 5 回 → 1 回 + 3 個の分岐デクリメントに変更。SIMD 化や autovectorization の前段で確実に効くマイクロ最適化。

**得た教訓:** LLVM は power-of-two な剰余を高速化するが、それ以外は実除算になる。リアルタイム DSP の inner loop では分岐デクリメント + 1 回だけ `%` がベター。

### 4.5 Windows + git bash の path 変換罠（再発）

`gen-params.mjs` の CLI entrypoint ガードで最初 ``import.meta.url === `file://${process.argv[1]}` `` と書いたが、Windows の `process.argv[1]` はバックスラッシュ込みパスで `import.meta.url` (`file:///C:/...`) と一致しない。`pathToFileURL(process.argv[1]).href` に修正。Phase 1 の `MSYS_NO_PATHCONV=1` パターンと同じ系統の罠。

### 4.6 Test の Phase 名 narration

Simplify レビューで「Phase 2 Step 14:」のような narration コメントは噛まない。仕様書 D-tag 参照（D17 / D29 等）は load-bearing なので残す。Phase 1/2 の比較が test の目的説明になっているケースも残す。判断軸:

- **削除**: コミット履歴 / step 番号 / 作業中の High 1/2 修正のような referencer
- **残す**: 仕様書 D-tag (`D29` 等)、Phase 1/2 比較で test の目的を説明している箇所、罠（write_index = 0 で初回 read がゼロ領域に当たる、等）の防壁解説

### 着手前に答えを出すべき調査項目

1. **Body Resonator 方式選択**: IR convolution (FFT-based、stable, 高品質) vs Modal Filter (IIR フィルタバンク、軽量) vs Static IR (固定 IR 1 本)。Phase 3 冒頭の pre-research で調査
2. **Thiran allpass for C8**: 1 次 Thiran で d ∈ [0,1] 補間、極が単位円内にあるかの安定性議論。Lagrange と比較した CPU コスト計測
3. **Loss filter design**: 弦の周波数依存減衰 (high-frequency damping) の 1-pole LPF or two-pole biquad。loop gain の安定性確認
4. **Pick position comb**: `feedforward delay z^-K` で comb 効果。fractional 化が必要か検討
5. **MIDI CC mapping アーキテクチャ**: SmoothedValue 経路で全ボイスに fan-out、`messages.ts` 拡張点
6. **UI voice meter の draw rate**: rAF (60Hz) で十分か、Worklet → main の round-trip API が必要か
7. **C ABI 互換性方針**: Phase 3 で関数追加するときの drift 対策 (check-wasm-exports REQUIRED 配列の管理)
