## 1. 概要

Phase 4b の主目的は「Phase 4a の 8 音ポリフォニック + LFO + Mod Wheel + Preset + 多楽器 7 種を土台に、**ピアノ音色を Stretching all-pass cascade (M=8 段、Rauhala-Välimäki 2006 closed-form) + Commuted impulse + velocity-dependent LPF (Hammer model) + Piano 用 Modal Body 係数で実装し `InstrumentKind::Piano = 7` を追加**」。補助的に `__synthDev.measureProcessTime` で F38b 計測自動化、`.gitattributes` で改行 LF 統一、`wasm-opt --metrics` で WASM サイズベースライン記録。

仕様書 18 ステップ・D56〜D67 の 12 設計判断を踏襲して 17 commits (うち 1 件は /simplify 後の cleanup commit) で実装、Phase 4a 互換性を `test_dispersion_disabled_matches_phase4a` で **Phase 4a HEAD (commit dfa81c3) との出力 256 frame × 2ch を ε=1e-6 でバイト一致機械保証**。

達成度サマリ: **F48 (`__synthDev.measureProcessTime`) は計測 API 整備完了、ユーザーによる実機 avg/max 値取得は持ち越し**。F49 (gzip 18.71 KB、目標 22 KB クリア) / F50 (Piano 0.047 ms / 非 Piano 0.029 ms、target 1.7 ms / 1.0 ms に対し圧倒的余裕) / F51-F58 cargo はすべて達成。**ただし Piano 音色の聴感は KS ベースの本質的限界により「弦楽器寄り」となり、本物のピアノとは異なる**。これは仕様書策定時から想定された制約 (§5 / §7 で詳述、Phase 4c で multi-string + Hertz hammer 等の構造拡張が候補)。

## 5. 既存コードの負債と次フェーズで再検討

- **Piano 音色が「弦楽器寄り」**: 本イテレーションの最大の負債。仕様書策定時から想定された KS ベースの構造的限界:
  - **KS ループは本質的に弦のシミュレーション** (1 voice = 1 遅延線)。Stretching all-pass で inharmonicity を加えても弦らしさは残る
  - **Hammer model が近似**: `note_on` 時のみの impulse + 1pole LPF。本物のフェルトハンマーは Hertz の非線形ばねで力–変位特性が時間と共に変化、独特の打撃 transient を生む。仕様書 §pre-research §6 で「Hertz law spring (Boutillon) や WDF は実装複雑度のため Phase 4c 送り」と明示
  - **複数弦 (1-3 弦 / note) の unison detuning なし**: 本物のピアノで A4 は 3 弦、unison のごく微小なピッチずれによる **beating + two-stage decay** がピアノらしさの根幹。Phase 4b は 1 voice = 1 弦
  - **Sympathetic resonance / damper 物理なし**: sustain ペダル時の他弦共鳴 (鍵盤の他の弦が振動) がピアノらしい余韻を生むが、Phase 4b では未実装
  - **Modal Body M=8 は粗い近似**: 実物の響板は数百 mode の応答を持つ。Phase 4b の biquad bank 8 個では構造ピーク 1-2 個しか再現できない
  - **Inharmonicity B が固定値**: Piano `B = 7.5e-4` は A4 基準。実物は低音ほど B 大 / 高音ほど B 小 (鍵盤 1〜88 で 10⁻⁴ → 10⁻³ オーダー)。`Ikey(f0)` 補正で多少追従するが、固定 B では低音の太さが出にくい
  - **聴感チューニング余地**: D62 で文献値 (Conklin 1996) ベースで実装したが、Piano プリセットの `damping` / `brightness` / `bodyWet` / `pickPosition`、Modal Body 係数の `gain` / `Q`、`HAMMER_CUTOFF_LOW/HIGH_PIANO` はいずれも聴感調整未実施。Phase 4c の主目的次第で「短期的に詰める方向」として再評価候補
- **F38b 実機計測の API は完成、計測値はユーザー操作で取得**: `window.__synthDev.measureProcessTime(durationMs)` を Console から呼び出す必要があるため Auto mode では完結不可。Phase 4c で CI / E2E ベースの自動計測 (Playwright + AudioWorklet 起動 + Console API 呼び出し) を検討する余地あり
- **`scripts/copy-wasm.mjs:34` の DEP0190 警告**: Phase 4a から継続、Phase 4b でも未対応。Phase 4c で `cross-spawn` 置換または別 Node API への移行を検討
- **TimingState の `as unknown as TimingState` cast 6 箇所**: production tree-shake 0 hits を実現するためのトレードオフ。`/simplify` レビューで「premature optimization」と指摘されたが実測ベースで必要と判断、cast の数を減らす方向の refactor は Phase 4c で再考可能
- **Piano プリセットの聴感 vs 仕様書 D62 文献値**: D62 で「文献値で初期実装、聴感調整は実装後」と明示されていたが、Phase 4b では聴感調整を行わなかった。Phase 4c で再評価
- **golden 値ファイル `tests/fixtures/phase4a_default_c4_v08.rs` の保守**: 527 行の f32 定数。将来 Phase 4a の挙動を変える場合は golden 再生成が必要 (worktree 経由の手順は §4 #10 で確立)

## 6. 開発フロー上の改善

- **18 ステップのうち F48 関連 (Step 14, Step 16) はユーザー操作必須で、Auto mode では計測値取得不可**。Phase 4a の F38b と同じ性質。仕様書策定時にこれを「先送り可能なステップ」として明示してから Auto 進行する判断は妥当
- **/simplify を retrospective 前に実行する習慣化**: コードレビュー (3 並列エージェント) が 6 件の改善 + 1 件の重複テスト削除を発見。Phase 4a では retrospective 直前に行わなかったため、`apply_instrument` 系の重複や rot コメントが Phase 4a まで残っていた可能性。Phase 4c でも /simplify を retrospective 前に組み込む
- **仕様書改訂 + 実装を同 commit で**: Phase 3 D36 案 D 追加、Phase 4a Triangle 式 typo、Phase 4b の `test_dispersion_b_zero_limit` 改訂、いずれも実装段階で発見。仕様書ドリブン開発の例外運用として「軽微な記述ミス / 数値判定基準の修正は実装と同 commit」が 3 イテレーション連続で発生、定石化
- **git worktree による golden 値取得パターン**: Phase 4b で確立 (§4 #10)。Phase 4a HEAD との byte 一致テストが必要な互換性保証で再利用可能
- **CRLF/LF 戦争は本当には終わっていない**: `.gitattributes` で LF 統一を宣言したが、Windows + git autocrlf の組み合わせで commit 時に warning が出続ける。blob hash 一致で実害はないが Phase 4c で `core.autocrlf=false` の明示推奨を README に追記する余地あり
- **聴感確認は仕様書とは別軸**: Phase 4b は cargo / clippy / 互換性テスト全 green でも「ピアノっぽくない」という結果に。**仕様書ドリブン開発の限界を経験**。Phase 4c で「実機聴感確認」を仕様書策定時の必須項目として組み込む候補

### 推奨スコープ (Phase 4c) — 重要度順

**主目的候補 (1 つ選定):**

1. **本格ピアノ音色** (Phase 4b の構造的限界を解消):
   - **Multi-string per voice** (1 voice = 3 弦結合 + unison detuning) — ピアノらしさの最重要要素
   - **Hertz law hammer (Boutillon)** — 非線形 spring で本物の打撃スペクトル
   - **Sympathetic resonance** — sustain ペダル時の他弦共鳴
   - **Soundboard モード増 (M=16)** + lid / room response
   - **Inharmonicity B(note) 関数化** — 低音 B 大 / 高音 B 小
   - 規模感: 25-30 step、Phase 4b の 18 step より大幅増、別 Engine 構造の可能性も
2. **C8 ピッチ自己発振モード** (Phase 1 から持ち越し、damping=1.0 経路 or FFT-based estimator)
3. **WASM SIMD** (`target-feature=+simd128`) — dispersion cascade 8 段を `f32x4` × 2 段で並列化候補

**補助候補 (Phase 4b で持ち越し中):**

4. **Pick position fractional 化** — Phase 4b は Piano kind では未使用、他楽器のリアリティ向上
5. **Look-ahead limiter** (5 ms 遅延、soft clip より透明)
6. **LFO 波形拡張** (S&H / Square / Sawtooth)
7. **LFO destinations 拡張** (Pick / Damping / BodyWet)
8. **楽器切替の fade-out / cross-fade** — Phase 4b D63 で撤回した「5 ms fade-out」を `PendingInstrumentChange` 状態機械で本実装
9. **Cross-tab preset 同期** (`storage` event)
10. **Preset JSON file import / export**
11. **Mono + Sustain 本実装** (Phase 4a / 4b でも no-op 継続)
12. **Piano プリセットの聴感チューニング** (Phase 4b 主目的に Piano を据える場合の短期改善)

### 調査項目 (Phase 4c 着手前に答えを出す)

- **本格ピアノを主目的にするか別軸 (SIMD / C8 自己発振) を選ぶか**: Phase 4b の Piano 音色が「弦楽器寄り」だった経験を踏まえ、Phase 4c でピアノを徹底的に追求する価値があるか / 別軸で完成度を上げるか
- **Multi-string per voice の Voice / VoicePool 設計**: 1 voice = N 弦結合の場合、`KarplusStrong` を `MultiStringPiano` 等の別 struct に分けるか、`KarplusStrong` 内に `n_strings: usize` を持たせるか。Voice trait の API 設計に影響
- **Hertz law hammer の per-sample コスト**: 非線形 ODE 統合は CPU 影響大、note_on 時のみで完結できるか / per-sample 必要か
- **Sympathetic resonance の Engine 構造**: damper 物理は全 voice 間の coupling が必要 (M:N 行列演算)、現状の VoicePool では表現困難。別 Engine 構造の検討
- **Modal Body M=16 への拡張**: WASM サイズ + CPU への影響 (現状 M=8 で 8 楽器 × 8 mode × 2ch = 128 biquad、M=16 で倍)
