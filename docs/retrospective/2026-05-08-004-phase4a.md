## 1. 概要

Phase 4a の主目的は「Phase 3 の 8 音ポリフォニック Karplus-Strong + Modal Body Resonator + Extended KS + MIDI CC + VoiceMeter 実装に **F38b 実機計測 / wasm-opt -O3 / `excitation_snapshot` cfg(test) で既存負債を解消** + **グローバル LFO + Mod Wheel master + プリセット保存 + 多楽器プリセット 6 種** を加える」だった。仕様書 17 ステップ・D44〜D55 の 12 設計判断を踏襲して 16 commits で実装、Phase 3 互換性を `test_mod_wheel_zero_disables_lfo` 等で機械保証。

達成度サマリ: **F38b は持ち越し継続 (操作習熟要因、cargo timing で代替確認)、その他 F39〜F47 はすべて達成**。F38b と F1〜F25 / F34 の Phase 1-3 持ち越し以外で未達なし。Mod Wheel = 0 の Phase 3 互換性、release cargo timing 0.023 ms / 128 frames (target 1.7 ms の 1.4% / 74× 余裕)、120 PASS + 1 IGNORED (regression なし) を確認。

## 4. 躓きと教訓

| # | 躓き | 回避策 / 教訓 |

| 1 | Windows + Node 24 で `node_modules/.bin/wasm-opt` を直接 execFileSync すると ENOENT | `process.platform === 'win32'` なら `.cmd` を最初に探し、`shell: true` で実行。`shell: true` は injection リスクの DEP0190 警告を出すが内部生成パスのみなので許容 |

| 3 | Triangle 式の仕様書 typo (式とテキストが矛盾) | 実装段階で発見、同 commit で仕様書側も修正。仕様書の数式は「テキスト記述 / コード例 / テスト期待値」の 3 軸で整合チェックすべき |
| 4 | `cargo fmt --check` がローカルで pass、CI で fail (rustfmt の長行展開判定差) | Phase 1-3 で確立した generator 出力には `#[rustfmt::skip]` を付与するパターンを Phase 4a で改めて適用。生成物の整形は generator 側で固定 |

## 5. 既存コードの負債と次フェーズで再検討

- **WASM gzip 18.42 KB が警戒ライン (18 KB) を 0.42 KB 微超過** (target 15 KB は超過、撤退 30 KB から余裕あり): `wasm-opt --print-stats` で重い pass を特定、楽器係数を 4 種に削減 (Sitar / Mandolin を Phase 4b 送り) すれば +1 KB 程度削減可能。Phase 4b で再評価
  - **Phase 4b Step 3 baseline (2026-05-09)**: `wasm-opt --metrics` で取得 (`--print-stats` は binaryen 旧オプション、現行版は `--metrics` / `--func-metrics`)。raw 40447 B / gzip **17784 B (17.37 KB)**。Internals: total 14539 instructions, funcs 40, exports 21 (18 C ABI + memory + tables/internal exports), globals 3, vars 243, memory-data 6013 B (主に楽器プリセット係数), Const 2760 / LocalGet 3938 / Store 1195 / Load 806 が物量。Phase 4b で BODY_MODES_PIANO_L/R + Piano 専用 const 追加 + dispersion.rs (closed-form 1 関数) で純増 ~0.6-1 KB 想定、目標 <22 KB / 警戒 <25 KB / 撤退 <30 KB
- **F38b 実機計測の手段が未整備**: `__synthDev` を release ビルドにも一時露出 + `performance.now()` で全 process 呼出の self time を計測する setInterval スクリプトを Phase 4b で実装し、ユーザー操作なしで avg/max が取れるようにする
- **`scripts/copy-wasm.mjs:34` の DEP0190 警告 (`shell: true`)**: Windows `.cmd` 実行のため許容、Phase 4b で `cross-spawn` への置換 or 別の Node API で代替できないか調査
- **prettier format 後の LF/CRLF 差分が毎 commit で発生**: `.gitattributes` で `* text=auto eol=lf` を明示してプロジェクト全体を LF 統一。Windows の git autocrlf を bypass。Phase 4b で実施
- **Voice State の Float32Array 再構築** (Phase 3 retrospective §5 から継続): `synth-processor.ts:198` の `voiceStateAmps` は constructor で確保済で実 alloc は 0、ただし SAB + Atomics で UI 同期する設計の方が clean。COOP/COEP 必須で GitHub Pages 不可のため Phase 5 以降
- **`gen-params.mjs` の整形ポリシー**: `#![allow(clippy::approx_constant)]` (module-level) と `#[rustfmt::skip]` (item-level) を生成側で出力するパターンが定着。生成器の出力をローカル / CI 双方の rustfmt と整合させるためには **新しい helper / const を追加する都度 fmt --check と CI 結果の両方を確認** する必要あり
- **`preset-schema.ts` と `messages.ts` の `LfoWaveformKey` / `InstrumentKindKey` 重複参照リスク**: 現状は messages.ts → preset-schema.ts の単方向 re-export で DRY、追加 enum 値が出ても messages.ts のみ変更で済む。Phase 4b で楽器追加時もこの方針を継続

## 6. 開発フロー上の改善

- **17 ステップのうち F38b 関連 (Step 1, Step 15) はユーザー操作必須で、Auto mode では自動完了不可**。これらを「先送り可能なステップ」として明示してから Auto 進行する判断は妥当 (Step 1 計測待ちでも Step 2-14 は依存しない実装で並行可能だった)
- **Step 6 で `synth.modWheel` を先行追加、Step 11 で `preset-schema.ts` の型のみ先行追加**: 仕様書の Step 順序が strict な依存関係を持たない場合、軽い前倒しで svelte-check / cargo build を pass できる。Phase 4b でも `WebAssembly` 設計判断と Web 側型定義に同様の循環依存が出る可能性あり
- **prettier format で意図しないファイルが LF/CRLF だけ整形される**: format → 関係ない変更を `git checkout` で revert → commit、を 5 回程度繰り返した。`.gitattributes` で改行統一すれば不要になる作業
- **CI rustfmt とローカル rustfmt の解釈差** (max-width 100 chars 判定): 生成物の整形は `#[rustfmt::skip]` で固定する Phase 1-3 のパターンを継続。新規 helper 追加時は CI 通過まで確認
- **コミット粒度**: 仕様書 Step = 1 commit を厳守、Step 17 (PR) で CI fail があり追加 commit が必要だったが、整然と二分探索可能な単位を維持できた
