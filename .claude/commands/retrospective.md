# Iteration Retrospective

直近のイテレーション（Phase / Sprint / MVP / マイルストーン）を振り返り、次イテレーションへ引き継ぐためのドキュメントを `docs/retrospective/<iteration-name>.md` に生成する。

## 引数

`$ARGUMENTS` がイテレーション名（例: `2026-05-06-001-mvp`、`phase2`、`v0.2.0`）。

引数が空の場合は以下から推定：

1. `docs/specs/` の最新ディレクトリ名（`ls docs/specs/ | sort | tail -1`）
2. それも無ければ git log の直近のマイルストーン的コミット範囲
3. 最後にユーザーへ確認を取る

## 事前収集

ドキュメントを書き始める前に、以下を収集する。**仕様書だけで書くと「やったこと」の薄い羅列になる。実装で得た学びを混ぜ込むこと。**

1. **仕様書全章**: `docs/specs/<iteration>/` 配下の `01-overview.md`〜`07-implementation-checklist.md` と `pre-research.md`（または相当ファイル）を Read
2. **コミット履歴**: `git log --oneline` で該当範囲を一覧。各コミットメッセージの主題を把握
3. **実装ファイル**: 主要パスを Read（仕様書からのズレ・追加判断を確認）
4. **テスト**: `cargo test` / `pnpm --filter ./web check` 等の現在の出力（必要なら実行）
5. **メトリクス**: WASM サイズ・ビルド時間・テスト数・ファイル数・依存関係数
6. **CI / デプロイ状態**: `.github/workflows/`、Pages 設定、branch protection
7. **既存メモリ**: `~/.claude/projects/.../memory/MEMORY.md` を Read。更新候補を洗い出す

## 生成セクション

以下の構成で `docs/retrospective/<iteration-name>.md` を Write する。

### 1. 概要

- 名前 / 期間 / 全体ゴール / 達成度 (X / Y)
- 1〜2 段落で「このイテレーションは何を残したか」

### 2. 達成と未達

- ✅ 達成したマイルストーン（チェックリスト粒度）
- ⚠️ 持ち越し / 未検証 / 次イテレーションへ送る項目（理由付き）

### 3. 主要な設計判断と振り返り

仕様書の設計判断（D1〜Dn 等）を 1 件ずつ再評価。各項目を：

- **採用してよかった** / **再検討が必要** / **未検証** にラベル
- 一言で根拠（実装した結果のフィードバック）

仕様書外で発生した判断（CI / hooks / lint config など）も含める。

### 4. 躓きと教訓

- 仕様書外で発生した問題と回避策
- 環境依存の罠（OS / shell / tool バージョン）
- 「次回も発生しうる」ものは特に明記

### 5. 既存コードの負債 / 次フェーズで再検討

- レビュー（Simplify 等）で「仕様書通り」として残した冗長
- 設計上のショートカット
- 二重管理 / モノフォニー前提 / 単一ファイル想定など、スケール時に壊れる箇所
- ファイルパスと行番号で参照

### 6. 開発フロー上の改善

- ツール設定（hooks / CI / branch protection / format / lint）
- 学んだプロセス（review rounds / commit granularity / etc.）

### 7. 次イテレーションへの引き継ぎ

- **必須前提**: 次に進む前に終わらせるべきこと（実機検証・修正・調査）
- **推奨スコープ**: 次イテレーションで何を入れるか / 何を別フェーズへ送るか（推奨と理由）
- **調査項目**: 着手前に答えを出すべき技術選定
- **参照点**: `path:line` 形式で 5〜10 箇所、次の人が最初に読むべきコード

### 8. メトリクス

| 指標 | 値 |
|---|---|
| コミット数 | |
| ファイル数（言語別） | |
| テスト数 | |
| WASM サイズ (gzip) | |
| ビルド時間 | |
| 仕様書ページ数 | |

### 9. メモリ更新案

このセッション後、auto-memory に追加・更新・削除すべき項目を提案。各項目に：

- 種別（user / feedback / project / reference）
- 名前（ファイル名提案）
- 1 行説明
- 採用/不採用はユーザーが判断

## 書き方の指針

- **読者は「未来の自分 / 別の人」**: そのまま判断材料にできる粒度で書く
- 仕様書を全文コピペしない。リンクと「ここを見ろ」で済ませる
- ❌ コミットメッセージのような「やったこと」だけで埋めない
- ✅ 「**なぜ** そうしたか」「**再現** すべきか」「**変更** すべきか」を書く
- 過大評価・過小評価を避け、淡々と事実と判断を分けて記述
- 仕様書からの逸脱があれば必ず理由とともに記載
- 検証できなかった項目（実機テスト等）は **未検証** と明示し、次イテレーションの前提条件にする
- 引き継ぎセクションは **次セッションのプロンプトとしてそのまま使える** くらい具体的に

## 出力後の挙動

1. ファイルを Write
2. ユーザーへ短い要約（このイテレーションの 3〜5 個の鍵となる学び、次イテレーションへの 1 行サマリ）を提示
3. メモリ更新案があれば、ユーザーに採否を確認してから保存
4. 「コミットしますか / PR を作りますか」とは **聞かない**。ユーザーが指示するまで working tree に置いておく
